# v1.7.1 Prerelease And Full Audit Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add SemVer prerelease release automation, update release-prep PR/docs, and complete a full-repository audit with targeted fixes before merging to `main`.

**Architecture:** Keep one release workflow for both stable and prerelease tags, deriving prerelease behavior from the tag format instead of duplicating packaging logic. Execute the repository audit as a structured review-and-fix pass across all modules, scripts, workflow files, and docs, landing only concrete risk-reducing cleanups and documentation improvements.

**Tech Stack:** GitHub Actions, PowerShell, Maven, Java 17, GitHub CLI, Swing, JUnit 5

---

### Task 1: Add Prerelease Tag Support To Release Automation

**Files:**
- Modify: `.github/workflows/release.yml`
- Test: local workflow logic review via shell inspection and release API reasoning

- [ ] **Step 1: Review current release trigger and tag parsing behavior**

Read `.github/workflows/release.yml` and identify all places that assume tags are only `vX.Y.Z`, especially:

- trigger patterns
- `TrimStart('v')` version extraction
- changelog section lookup
- GitHub release creation inputs

- [ ] **Step 2: Define prerelease-compatible tag matching**

Update workflow trigger patterns so both stable and prerelease tags are accepted, for example:

- stable: `v1.7.1`
- prerelease: `v1.7.1-rc1`

Avoid broad catch-all patterns that would match non-SemVer operational tags.

- [ ] **Step 3: Add explicit tag parsing for release metadata**

In `.github/workflows/release.yml`, compute:

- raw tag name from `GITHUB_REF_NAME`
- effective version for changelog lookup by removing only the leading `v`
- prerelease flag by checking for a SemVer suffix after the numeric core version

Keep this logic in a dedicated PowerShell block so it is readable and reusable later in the workflow.

- [ ] **Step 4: Preserve artifact naming and packaging behavior**

Keep portable ZIP, installer `.exe`, and `.exe.sha256` generation unchanged aside from allowing prerelease version strings to flow through filenames.

- [ ] **Step 5: Mark GitHub releases as prereleases automatically**

Update the `softprops/action-gh-release` step to set `prerelease: true` when the derived prerelease flag is true, and false otherwise.

- [ ] **Step 6: Re-read the workflow and verify stable + prerelease paths**

Manually inspect the resulting YAML and confirm:

- stable tags still produce normal releases
- prerelease tags produce prereleases
- changelog lookup still resolves the correct heading
- artifact paths still match generated outputs

- [ ] **Step 7: Commit**

```bash
git add .github/workflows/release.yml
git commit -m "build(release): add semver prerelease tag support"
```
