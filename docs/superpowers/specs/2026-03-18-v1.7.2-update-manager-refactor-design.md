# v1.7.2 Update Manager Refactor Design

Date: 2026-03-18
Target Version: 1.7.2
Status: Draft for review

## Summary

Refactor the application update flow into a dedicated update manager that:

- tracks stable and prerelease releases from GitHub
- supports persistent prerelease opt-in
- reports current version, latest stable, latest experimental, and update distance
- distinguishes between soft and hard update notifications
- supports guided upgrade and downgrade to any discoverable GitHub version
- keeps the GUI header/footer version aligned with the running build
- improves installer UX with visible install/update progress and automatic relaunch

This work will ship as `1.7.2` on a fresh branch from `main`. The retired `dev` branch is not part of the new delivery model.

## Goals

- Replace the current single-endpoint "latest release" logic with a release catalog model.
- Always check for the latest stable release.
- Optionally check prerelease builds when the user enables a persistent experimental-updates setting.
- Show the operator:
  - currently installed/running version
  - latest stable version
  - latest experimental/prerelease version
  - whether the current version is current, one stable release behind, or behind a newer stable line
  - how many stable releases behind the current version is
- Allow upgrade or downgrade to any discoverable GitHub release that has a valid installer and checksum.
- Warn before installing a prerelease or downgrading to an older version.
- Show a persistent toolbar warning for meaningful available updates even if the startup popup is ignored.
- Ensure the displayed GUI version reflects the actual running build.
- Make installer/update activity visible and relaunch the app automatically after success.

## Non-Goals

- Differential patch updates
- background auto-install without user confirmation
- rollback snapshots or automatic state migration
- a second long-lived integration branch

## Architecture

## Update Catalog

Add a release catalog layer under `core.update` that fetches GitHub releases from the releases list API instead of relying on `/releases/latest`.

Responsibilities:

- load release history from GitHub
- normalize tags and release metadata
- classify releases as stable or prerelease
- retain only actionable releases for guided install when required assets exist
- expose the full version list for manual version selection

Suggested classes:

- `UpdateCatalogService`
- `UpdateCatalog`
- `CatalogRelease`
- `CatalogReleaseType`

## Update Policy

Add a policy layer that interprets release data relative to the running version.

Responsibilities:

- identify latest stable release
- identify latest prerelease release
- count how many stable releases behind the current version is
- determine whether the current version is:
  - current
  - one stable release behind in the same release line
  - behind a newer stable release line
- determine whether to show:
  - no prompt
  - soft recommendation only
  - hard startup prompt plus toolbar badge
- track ignored hard-prompt target versions on a per-target basis

Suggested classes:

- `UpdatePolicyService`
- `UpdateStatus`
- `UpdatePromptPolicy`

## Update Actions

Add an action layer for selecting and validating install targets.

Responsibilities:

- list valid install targets from catalog data
- support upgrade and downgrade
- enforce confirmation warnings for:
  - prerelease installs
  - downgrades
- reuse guided installer download and checksum verification

Suggested classes:

- `UpdateActionService`
- `InstallTarget`
- `InstallWarningPolicy`

## Version Resolution

Unify runtime version resolution so GUI and CLI both display the actual running version consistently.

The version shown in:

- window header
- footer version label
- updater current-version status

must come from a shared runtime-version resolver that prefers packaged runtime metadata and only falls back to embedded resources when necessary.

Suggested classes:

- extend `VersionSupport` or add `RuntimeVersionService`

## GUI Behavior

## Settings

Extend the GUI settings area to include a persistent update-management section.

Display:

- current installed/running version
- latest stable version
- latest experimental/prerelease version
- update status summary
- stable updates behind count

Controls:

- persistent checkbox: enable experimental/prerelease updates
- check updates now
- choose version to install

## Startup Prompting

Stable update behavior:

- if the current version is one stable release behind in the same line:
  - no blocking startup prompt
  - show soft "update recommended" messaging
- if the current version is behind a newer stable release line:
  - show a startup prompt every launch
  - allow the user to ignore that exact target version
  - continue showing the toolbar warning badge even after ignore

Ignore behavior:

- ignore applies only to the exact prompted target version
- future target versions must prompt again

## Toolbar Warning

When a meaningful stable update is available, show a warning icon in the toolbar or tools/navigation area.

Rules:

- must remain visible even if the popup was ignored
- should clear only when:
  - the current version is no longer behind that meaningful target
  - release state changes such that the prompt is no longer relevant

## Manual Version Picker

Provide a version picker that lists all discoverable GitHub releases.

Each entry should show:

- version tag
- stable/prerelease marker
- release title if useful

Behavior:

- allow upgrade to newer versions
- allow downgrade to older versions
- allow prerelease selection if the user has opted into experimental updates, or explicitly chooses to browse all available versions if product design prefers broader visibility
- before installing an older or experimental version, show a warning dialog that features may be missing or broken

## Release and Comparison Rules

The stable track is always active.

The experimental/prerelease track is active only when the persistent setting is enabled.

Comparison rule:

- one stable release behind in the same release line means "up to date enough, update recommended"
- moving from one release line to a newer one, such as `1.7.x` to `1.8.0`, means "new update available" and triggers the hard prompt flow

For `updates behind`, count stable releases only.

Prereleases are informational and selectable, but they do not replace the stable-track policy.

## Installer and Update UX

Installer-driven install/update flows should always show visible progress instead of appearing silent.

Required behavior:

- always show a small status window during install/update
- use the same experience for:
  - manual `.exe` launch
  - in-app updater launch
- show phases such as:
  - preparing update
  - uninstalling or replacing old version
  - installing new version
  - finishing setup
  - launching application
- on success:
  - automatically relaunch the newly installed app
- on failure:
  - keep the status window open
  - show the error and next-step guidance

In-app initiated updates may provide richer context text, but the install UX should not depend on the GUI being open.

## Risks

- release ordering must be deterministic for stable and prerelease tags
- downgrade support increases user-facing risk if configuration or expectations changed between versions
- GitHub asset naming must stay consistent with checksum matching rules
- install/relaunch behavior must avoid double-launch or stale-process edge cases
- hard-prompt suppression must remain scoped to the exact ignored target version

## Acceptance Criteria

- latest stable and latest prerelease are both discoverable through the update manager
- prerelease discovery is controlled by a persistent GUI setting
- GUI shows current, latest stable, latest prerelease, and stable updates-behind count
- one stable release behind in the same line produces recommendation-only behavior
- newer stable release line produces startup prompt plus persistent toolbar warning
- ignoring a prompt suppresses only that target version popup
- guided install can target any valid release version, including downgrades
- downgrade and prerelease installs require warning confirmation
- GUI header/footer version matches the actually running build
- installer/update flow shows visible progress and relaunches the app after success
