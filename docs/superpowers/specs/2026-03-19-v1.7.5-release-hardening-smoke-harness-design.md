# v1.7.5 Release Hardening Smoke Harness Design

## Goal

Add a release-hardening smoke harness that verifies the application's critical workflows end to end before shipping, with shared coverage across repo/JAR and packaged-app execution modes.

## Scope

This patch focuses on functional release confidence, not broad architectural cleanup. It establishes a deterministic smoke layer and coverage inventory so future cleanup and refactor work can be prioritized against real coverage gaps.

In scope:

- Shared smoke scenario catalog
- Repo/JAR smoke runner
- Packaged-app smoke runner
- Coverage matrix for GUI workflow parity
- Release-facing smoke documentation
- ADR for smoke boundaries and printer validation policy

Out of scope:

- Full Swing UI automation
- Whole-codebase SRP/DRY/SOLID refactors
- Broad algorithm rewrites without harness evidence

## Architecture

The hardening layer should use one shared scenario definition and two execution targets:

1. Repo/JAR mode for fast developer verification before packaging
2. Packaged-app mode for release verification against the actual distributed artifact

Both modes should execute the same scenario definitions, emit a single pass/fail report, and validate identical expected outcomes wherever possible.

GUI workflows should be validated through shared backend services or CLI-equivalent paths first rather than trying to automate Swing interactions in this patch. Any GUI action without a smokeable backend path must be explicitly documented as a coverage gap instead of being silently assumed covered.

## Coverage Model

### Tier 1: Must-Pass Release Scenarios

- `config`
- `db-test`
- Shipment label generation in print-to-file mode
- Carrier-move label generation in print-to-file mode
- Rail preview/render generation
- System-default rail print path validation without sending live print jobs
- Packaged config asset resolution:
  - `walmart-sku-matrix.csv`
  - location matrix
  - printer YAML files

### Tier 2: Environment-Sensitive Checks

- Printer reachability by ping and TCP `9100`
- Tropicana per-user config resolution
- Packaged installer/bootstrap install validation

### Tier 3: Audit-Only Gaps

- GUI options with no CLI/shared-service equivalent yet
- Any flow still requiring manual confirmation

## Implementation Shape

### Smoke Manifest

Add a smoke manifest that defines:

- scenario name
- target modes
- required inputs
- command or service entrypoint
- expected exit status
- expected generated artifacts
- optional validation hooks

### Shared Harness

Add a PowerShell smoke harness that:

- resolves runtime config source
- selects repo or packaged target mode
- executes manifest scenarios
- captures exit code and high-signal output
- validates expected artifacts and key invariants
- writes both human-readable and machine-readable reports

### GUI Coverage Mapping

Add a coverage matrix that maps:

- GUI action
- backing class/service
- smoke path
- status: `covered`, `partial`, or `gap`

This matrix is part of the release safety boundary. Missing parity should be explicit.

### Small Smoke Entry Points

Add small smoke-only entrypoints only where necessary to expose existing shared backend behavior for GUI-only workflows. Do not duplicate workflow logic in test scripts.

## Guardrails

- Smoke printing must not send live jobs unless explicitly requested.
- Printer validation defaults to reachability checks, not file submission.
- Production WMS use remains read-only and should use a bounded set of known smoke IDs.
- Harness should fail fast on:
  - missing config
  - missing packaged assets
  - missing default printer where required
  - mismatched or missing output artifacts

## Documentation and Governance

Add:

- this spec

- ADR documenting the smoke boundary

Update:

- `README.md` with smoke commands, config expectations, and coverage boundaries
- release instructions/checklist so release tagging requires smoke evidence

## ADR Intent

The ADR should capture:

- why release gating must cover both repo and packaged modes
- why GUI coverage is validated through shared backend paths first
- why printer smoke checks default to reachability instead of live print jobs

## Recommended Outcome

Implement this as the next `1.7.5` hardening patch. After the harness lands, use its coverage report to drive the next documentation, cleanup, and refactor patches with evidence rather than intuition.
