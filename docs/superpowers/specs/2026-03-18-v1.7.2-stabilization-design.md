# v1.7.2 Stabilization Design

Date: 2026-03-18
Target Version: 1.7.2
Status: Draft for review

## Summary

Stabilize the in-progress `v1.7.2` update-manager branch by addressing the concrete review findings that affect correctness, upgrade safety, installer behavior, configuration semantics, data-layer efficiency, and operator-facing polish.

This pass is not a new feature wave. It is a focused hardening pass on top of the existing `v1.7.2` work so the branch can ship with:

- correct SemVer prerelease ordering
- complete GitHub release catalog coverage for manual downgrade selection
- prerelease visibility that respects the persistent experimental opt-in setting
- prerelease-aware installer helper behavior
- a valid installer status window lifecycle on failure and success
- explicit printer-routing rule semantics instead of partial silent support
- lower query amplification during shipment SKU description resolution
- cleaned-up operator-facing documentation and settings copy

The work should be tracked in new GitHub stabilization issues under `v1.7.2` so commit history can reference the follow-up fixes separately from the original update-manager implementation.

## Goals

- Fix confirmed updater correctness bugs:
  - prerelease ordering for numeric suffixes such as `rc2` vs `rc10`
  - release catalog pagination so older downgrade targets remain visible
  - prerelease install-target visibility so experimental targets are hidden unless opt-in is enabled
- Fix confirmed installer-helper bugs:
  - prerelease installer filename parsing
  - invalid status-window failure handling
- Clarify printer-routing configuration semantics by either fully supporting `when.all` or rejecting unsupported multi-condition rules explicitly.
- Reduce avoidable N+1 description lookup behavior in `OracleDbQueryRepository` during shipment footprint loading.
- Clean up directly related GUI and docs issues:
  - README workflow badges
  - settings dialog copy
  - small DRY cleanup in the guided updater flow
- Add regression coverage for each confirmed bug where practical.

## Non-Goals

- broad architectural rewrites unrelated to the confirmed findings
- redesign of the full update manager UX beyond the fixes above
- replacing the installer technology or packaging model
- unrelated database query rewrites outside the shipment-footprint description path

## Architecture

## Updater Correctness

Keep the existing `v1.7.2` update-manager architecture, but tighten the correctness boundaries:

- `VersionSupport` remains the shared version utility, but its comparison logic must become numeric-aware for prerelease token fragments.
- `UpdateCatalogService` remains the release ingestion point, but it must iterate through GitHub release pagination rather than assume one page is sufficient.
- `UpdateActionService` remains responsible for install target selection, but install-target visibility must take the persistent experimental setting as an explicit input instead of leaking prerelease targets by default.

The behavior target is simple:

- stable releases are always visible to policy logic
- prerelease metadata may be visible for status purposes
- prerelease install targets are only selectable when experimental updates are enabled

## Installer Helper

Keep the helper-script-based install handoff, but make its state flow valid and prerelease-aware.

Required changes:

- parse stable and prerelease version strings from installer filenames
- keep a single visible status window alive for the helper lifecycle instead of mixing modeless and modal display APIs
- show final success or failure state predictably
- preserve relaunch behavior after successful helper-driven updates

This should improve the in-app guided install path without changing the raw Windows installer UX when users run the packaged installer directly.

## Routing and Data Layer

Two correctness/maintainability changes should land together because they both prevent misleading behavior:

- `PrinterRoutingService` should stop silently honoring only the first entry of `when.all`
- `OracleDbQueryRepository` should stop probing description tables one SKU at a time when shipment footprint rows can be resolved more efficiently in batch or per-request cache form

For routing, the preferred option is to implement true `all` semantics if the config model already presents that contract cleanly. If that would create disproportionate risk, the fallback is to reject multi-condition rules with an explicit validation error instead of silently discarding extra conditions.

For the DB layer, the goal is not a full repository rewrite. The goal is to collapse repeated description lookups enough that large shipments do not pay a separate `PRTDSC` and `PRTMST` search cost per row.

## GUI and Documentation

This pass should make only focused UI changes tied to the reviewed findings:

- correct README branch badges to `main`
- fix the settings dialog copy typo and related wording
- reduce small duplication in the guided update flow
- keep broader `LabelGuiFrame` refactors targeted to the update/install responsibilities touched by this stabilization work

The bar for refactoring is risk reduction and clearer responsibility boundaries, not cosmetic rearrangement.

## Testing

Add or expand tests for:

- prerelease comparison ordering:
  - `rc2 < rc10`
  - stable releases remain higher precedence than prereleases of the same base version
- paginated release catalog loading
- install-target gating when experimental updates are disabled
- prerelease installer filename parsing
- installer-helper negative-path state handling where practical
- routing validation or multi-condition matching, depending on chosen implementation
- shipment description lookup behavior if the DB refactor introduces a new helper or cache boundary

Retain existing targeted module tests and rerun broader module tests after the fixes land.

## Verification

Minimum verification for this stabilization pass:

- targeted updater tests
- targeted GUI tests
- targeted DB tests if new ones are added
- full module test suite:
  - `.\mvnw.cmd -q -pl core,db,gui,cli -am test`
- packaging validation as needed for helper-script changes:
  - `powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\build-jpackage-bundle.ps1 -InstallerType exe`

If installer-helper behavior changes materially, rerun the checksum validation on the produced installer artifacts.

## Risks

- GitHub pagination introduces more network and parsing logic, so guardrails are needed to avoid infinite paging or excessive requests.
- SemVer fixes must remain compatible with existing stable-only comparisons.
- Routing behavior changes can expose previously hidden misconfigurations; that is acceptable if the failure is explicit and actionable.
- Data-layer optimizations must preserve current fallback order for description resolution:
  - `PRTDSC`
  - `PRTMST`
  - fallback shipment/order description

## Delivery

Sequence:

1. create the stabilization GitHub issues and labels if missing
2. update this spec with issue references if needed
3. write the implementation plan
4. implement with TDD where feasible
5. commit in small conventional commits that reference the stabilization issues
6. rerun verification
7. request code review before updating the `v1.7.2` PR

Example commit shapes:

- `fix(update): correct prerelease ordering and release pagination (#NN)`
- `fix(installer): support prerelease installer parsing (#NN)`
- `refactor(db): reduce shipment description query amplification (#NN)`
- `docs(readme): point workflow badges at main (#NN)`
