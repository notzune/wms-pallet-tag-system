# v1.7.1 Prerelease And Full Audit Design

## Goal

Add first-class GitHub prerelease support using SemVer prerelease tags such as `v1.7.1-rc1`, and perform a full-repository audit before merging release-prep work to `main`.

## Scope

This work covers two coordinated tracks:

1. Release-process changes
2. Full codebase and documentation audit

The release-process track includes:

- updating the GitHub Actions release workflow to trigger on prerelease tags
- marking GitHub Releases as prereleases automatically when the tag contains a SemVer prerelease suffix
- preserving the existing artifact contract for portable ZIP, installer `.exe`, and installer `.exe.sha256`
- updating operator/developer docs and the open PR metadata to reflect the prerelease flow

The audit track includes review of the entire tracked codebase:

- `cli`
- `core`
- `db`
- `gui`
- `scripts`
- `.github/workflows`
- top-level docs such as `README.md`, `CHANGELOG.md`, and release notes / handoff docs
- Javadocs and every `package-info.java`

## Non-Goals

This work does not introduce a separate updater channel, a second packaging stack, or silent/in-place self-updating. It also does not force broad architecture churn when no concrete bug, maintainability problem, or risk justifies the change.

## Release Workflow Design

The repository should support two tag classes:

- stable tags: `vX.Y.Z`
- prerelease tags: `vX.Y.Z-<suffix>` such as `v1.7.1-rc1`

One workflow should continue to handle both paths. The workflow will:

- trigger on both stable and prerelease SemVer tags
- derive the effective version from the tag for packaging and changelog lookup
- determine whether the tag is a prerelease by checking for a SemVer suffix
- create a GitHub Release with `prerelease: true` only for prerelease tags
- continue publishing the same artifacts for both stable and prerelease runs

This keeps packaging logic unified and avoids stable/prerelease drift.

## PR And Documentation Design

The current release-prep PR should be retained rather than closed and recreated. Its title/body should be updated to reflect that it is preparing `v1.7.1` for prerelease and final release flow.

Documentation should define the expected sequence:

1. Merge the release-prep PR to `main`
2. Tag `v1.7.1-rc1` to publish a GitHub prerelease
3. Validate release artifacts and updater behavior against the prerelease
4. Fix any issues if needed
5. Tag `v1.7.1` for the final release

README and any release-process notes should explicitly state that prerelease tags publish GitHub prereleases automatically.

## Full Audit Design

The audit is repository-wide, not hotspot-limited. Larger or historically riskier classes still receive extra attention, but every production area is reviewed.

The audit should evaluate:

- correctness and edge-case handling
- release/update path reliability
- algorithmic inefficiencies and repeated scans/allocations
- SRP, DRY, and SOLID adherence where it materially affects maintainability or bug risk
- test coverage gaps around behavior that can regress silently
- clarity and completeness of Javadocs
- completeness and accuracy of `package-info.java`
- operator/developer documentation consistency with actual behavior

Findings should be sorted into three categories:

- must-fix before merge
- low-risk improvements worth landing now
- larger follow-up refactors or documentation tasks to defer

## Constraints And Review Standard

Changes should remain pragmatic:

- fix real issues, not theoretical style concerns
- preserve working behavior unless a change is intentional and documented
- keep refactors bounded to the problem they improve
- avoid unnecessary subsystem churn during release-prep unless the current state is risky enough to justify it

## Validation Plan

Validation should include:

- workflow/release logic verification for stable and prerelease tags
- targeted tests added or updated for prerelease release behavior
- packaging validation for app-image and installer output
- artifact checksum verification
- selective build/test reruns driven by audit findings
- final documentation review to ensure release flow and code comments match shipped behavior

## Expected Outcome

After this work:

- prerelease tags such as `v1.7.1-rc1` can publish GitHub prereleases automatically
- the current PR clearly represents release preparation rather than an ad hoc one-off
- the repository has completed a full audit pass with concrete fixes and documented deferred items
- release documentation and Javadocs better match the real system behavior
