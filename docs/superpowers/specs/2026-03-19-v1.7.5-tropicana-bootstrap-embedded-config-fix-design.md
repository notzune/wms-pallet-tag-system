# v1.7.5 Tropicana Bootstrap Embedded Config Fix Design

**Date:** 2026-03-19

## Goal

Ship `1.7.5` as a narrow patch release that fixes the broken embedded-config handoff in the Tropicana bootstrap/support installer flow and adds end-to-end regression coverage so the same failure cannot ship again.

## Problem

`1.7.4` shipped a Tropicana bootstrap EXE and generated fallback `Install-Tropicana-Config.ps1`, but the generated support script did not activate its embedded config payload. The script therefore fell back to searching for an external `.env` file that was not present in the one-file distribution path.

Operationally, that means:

- the packaged app installs
- the Tropicana per-user config file is never written
- the app continues running against the public dummy config or no Tropicana config at all
- workflows such as Rail Labels fail with missing config keys or DB/auth errors

## Root Cause

The generated `Install-Tropicana-Config.ps1` template uses an embedded base64 payload placeholder. After template substitution, the script still compares the payload variable against the substituted literal instead of a stable sentinel token. That condition never becomes true, so the runtime parameter `ConfigContentBase64` remains empty and the embedded payload is ignored.

This is a packaging-script regression, not a GUI workflow bug and not a runtime config-discovery bug.

## Constraints

- Keep this within patch SemVer scope.
- Do not redesign public packaging or Tropicana distribution.
- Keep Tropicana config external to the install directory.
- Keep `1.7.4` published, but document that `1.7.5` fixes the broken Tropicana bootstrap path.

## Selected Approach

1. Fix the embedded-payload activation logic in `Install-Tropicana-Config.ps1` with a stable sentinel constant.
2. Keep `build-tropicana-installer.ps1` generating and shipping the same fallback script, but test that generated artifact end to end.
3. Strengthen PowerShell regression tests so they execute the generated support script without `-ConfigSourcePath` and verify a real `%LOCALAPPDATA%` config file is written.

## User Flow

### Before fix (`1.7.4`)

- User runs the Tropicana setup EXE.
- App installs.
- Post-install config step silently fails to apply embedded config.
- User opens the app and gets missing-config or DB failures.

### After fix (`1.7.5`)

- User runs the Tropicana setup EXE.
- App installs.
- Embedded fallback support script writes `%LOCALAPPDATA%\Tropicana\WMS-Pallet-Tag-System\wms-tags.env`.
- App opens and uses the installed Tropicana config successfully.

## Technical Design

### Embedded payload activation

- Introduce a stable placeholder token like `__EMBEDDED_CONFIG_BASE64__`.
- Keep the template check against that exact sentinel, not against the rendered payload text.
- When the rendered script sees a non-sentinel embedded payload and no explicit `-ConfigContentBase64` parameter, it should automatically load the embedded content.

### Regression coverage

- Keep the existing support-script test for direct `-ConfigSourcePath` behavior.
- Add a stronger generated-artifact test in `scripts/tests/build-tropicana-installer.tests.ps1`:
  - generate support script from a temp payload
  - run generated script with `-LocalAppDataRoot` and `-InstallDir` but without `-ConfigSourcePath`
  - verify written file contents match the embedded payload

### Release documentation

- Bump project version to `1.7.5`.
- Add changelog note that `1.7.4` Tropicana bootstrap/support installer had a broken embedded-config handoff and `1.7.5` fixes it.

## Testing Strategy

- Unit/regression tests for support script behavior.
- Generated-artifact execution test for the built fallback script.
- Local Tropicana bootstrap build smoke.
- Manual app-image config smoke to confirm `%LOCALAPPDATA%` config is selected after support-script install.

## SemVer

Patch release. This fixes a shipped installer/configuration regression without changing public APIs or broad product behavior.
