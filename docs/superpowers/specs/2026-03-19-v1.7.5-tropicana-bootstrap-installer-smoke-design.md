# v1.7.5 Tropicana Bootstrap Installer Smoke Design

## Goal

Close the remaining release-critical smoke gap for the Tropicana bootstrap installer by adding a real local install validation path that proves:

- the Tropicana bootstrap artifact can be built locally
- the installer can install the app into a real temp location
- the install writes Tropicana per-user config to `%LOCALAPPDATA%\Tropicana\WMS-Pallet-Tag-System\wms-tags.env`
- the installed app resolves that config instead of the bundled template
- the installed app can pass `config` and `db-test`

This is release hardening for `1.7.5`, not a packaging redesign.

## Current State

- The release smoke harness already covers repo and packaged app-image paths.
- The coverage matrix still marks `Tropicana bootstrap installer flow` as `partial`.
- `build-tropicana-installer.ps1` builds an IExpress bootstrapper that wraps:
  - the packaged jpackage installer
  - `Install-Tropicana-Config.ps1`
- The current installer helpers only launch the packaged installer with `/log`.
- A direct probe showed that running the packaged installer with `/?` opened UI and did not expose a fast unattended help path.

## Scope

In scope:

- add a release-only installer smoke scenario to the smoke manifest
- extend the smoke runner to optionally execute installer scenarios
- verify real local installation into a temp path
- verify Tropicana config creation and runtime precedence after install
- document the installer smoke requirement in the release docs

Out of scope:

- full Swing GUI automation
- live printer-job submission
- unrelated refactors
- changing the public-vs-Tropicana packaging boundary

## Approach Options

### Option 1: Always run full installer smoke

Run the Tropicana bootstrap installer on every smoke invocation.

Pros:

- strongest default safety

Cons:

- too slow for normal development
- higher flake risk during fast feedback loops

### Option 2: Installer smoke only through fallback support script

Skip the actual EXE installer and only validate `Install-Tropicana-Config.ps1`.

Pros:

- easy to automate

Cons:

- does not cover the real operator path
- leaves the release-critical bootstrap gap open

### Option 3: Release-only real installer smoke

Keep the fast smoke path unchanged and add an opt-in installer scenario for release validation.

Pros:

- covers the real operator path
- preserves fast repo/package smoke for day-to-day work
- fits release gating naturally

Cons:

- requires installer automation support

Recommendation: Option 3.

## Design

### Smoke Manifest

Add a packaged-only scenario for Tropicana bootstrap install, marked as release-only.

The scenario should express:

- builder command requirement
- installer artifact expectation
- installed app verification commands
- Tropicana config path expectation

Default smoke runs should skip this scenario unless explicitly enabled.

### Smoke Runner

Extend `scripts/run-smoke-tests.ps1` with an opt-in switch such as `-IncludeInstallerScenarios`.

When enabled, the runner should:

1. Build the Tropicana bootstrap installer locally.
2. Create isolated temp roots for:
   - install location
   - `%LOCALAPPDATA%`
   - installer logs
3. Run the bootstrap installer in unattended or automation-friendly mode.
4. Wait for install completion with a bounded timeout.
5. Assert the Tropicana config file exists in the temp `%LOCALAPPDATA%`.
6. Run the installed app `config`.
7. Run the installed app `db-test`.
8. Fail clearly on any missing file, bad exit code, timeout, or config-precedence regression.

### Installer Automation Constraint

This design does not assume the current packaged installer already supports unattended install.

Implementation must first verify whether the jpackage EXE can be driven non-interactively with supported arguments. If not, the patch must add a safe automation seam, likely by:

- extending the bootstrap wrapper to support a non-interactive install path for smoke
- or providing a release-smoke-only launcher path that installs from the packaged app image into a temp root while still exercising the bootstrap orchestration

The smoke harness must not fake success by skipping the install step.

### Evidence Output

The smoke report should include:

- bootstrap installer build result
- installer exit code or timeout
- installed app path
- resolved config file path from the installed app
- `db-test` result

## Failure Policy

The installer scenario fails if:

- the Tropicana EXE is not built
- the installer hangs or exits non-zero
- `%LOCALAPPDATA%` Tropicana config is missing
- the installed app resolves the bundled template instead of the per-user Tropicana config
- `db-test` fails after install
